# 從對話擷取技能

這個技能的目的是把目前對話中產生的知識、模式與工作流程整理下來，並保存為可重複使用的技能。

## 何時使用

- 使用者說「把這個整理成 skill」或「這個留到下次用」
- 對話中產生了實用的流程、模式或領域知識
- 使用者希望把新學習更新到既有技能
- 對話挖掘出值得保留的不直觀步驟、常見陷阱或最佳實務

## 先做分診（本工作區強制）

動手前先回答 `dev-diagnosis` 的第一問：**已經有類似的 skill 了嗎？**
- 查 `~/.claude/skills/` 是否有同類（特別注意 `skillify`、`huashu-nuwa`、`writing-skills`，與 `skill-builder` agent）
- 若已有 → 走「更新既有技能」分支，不要重複造
- 若無 → 才新增

## 擷取流程

### 階段 1：辨識要擷取的內容

回顧整段對話，找出以下資訊：

- **工作流程（Workflows）**：經過多次嘗試才收斂出的多步驟流程
- **領域知識（Domain knowledge）**：不直觀、但關鍵的事實、設定或限制
- **陷阱與修正（Gotchas and fixes）**：遇到的問題與可行解法
- **可重用模式（Patterns）**：有效的程式寫法、指令序列或模板
- **決策理由（Decision rationale）**：為何選 A 不選 B

先摘要你打算擷取的內容，並在動手前向使用者確認。

### 階段 2：決定保存位置

如果使用者已經指定技能名稱或保存位置，或從上下文可明確推斷，就直接執行。否則，使用 `AskUserQuestion` 工具（或自然對話）釐清：

- **新增技能還是更新既有技能？**
- **若更新既有技能**：要更新哪一個技能？（列出 `~/.claude/skills/` 內相關技能）
- **若新增技能**：技能名稱應該叫什麼？
- **保存位置（本工作區慣例）**：使用者層級 skill 放 `~/.claude/skills/<name>/SKILL.md`（跨專案共用）；若要與某專案綁定，放該 repo 的 `.claude/skills/<name>/SKILL.md`

### 階段 3：撰寫技能內容

#### 新增技能時

- 選擇清楚易懂的名稱（小寫、連字號、最多 64 字元）
- 撰寫具體 description，清楚說明「做什麼」與「何時使用」（第三人稱），並嵌入觸發詞
- 將對話蒸餾成精簡、可執行的操作指引
- 放入來自對話的具體範例
- 補上實際用到的工具腳本或指令

#### 更新既有技能時

- 先閱讀現有 `SKILL.md`
- 判斷新內容適合放在哪裡（新增段落、更新步驟、補充範例）
- 整合時避免重複既有內容
- 保持原本的結構與語氣

### 階段 4：蒸餾準則

目標是把雜亂對話轉成乾淨、可重用、可執行的指引。

**應該做：**

- 擷取最終可行解法，不是失敗嘗試（除非失敗本身是重要陷阱）
- 由具體案例抽象化（把硬編碼值改成 placeholders）
- 補上不直觀步驟背後的原因
- 加入只有透過這次對話才會知道的關鍵上下文
- 全文控制在 500 行以內

**不要做：**

- 不要保留對話痕跡（例如「如同我們剛才討論」）
- 不要重複 agent 本來就知道的資訊
- 不要塞入無法遷移到其他情境的過度細節
- 程式範例已足夠時，不要再寫冗長說明

### 階段 5：寫入與驗證

- 建立或更新對應技能檔案
- 確認 `SKILL.md` 在 500 行以內
- 檢查 description 是否具體，且包含觸發詞
- 向使用者確認擷取結果是否正確

### 階段 6：登記（本工作區強制）

- 新增 skill 後，到 `dev-diagnosis-skill/stations/claude-workspace.md` 的 skill 清單補一筆，避免下次有人重造
